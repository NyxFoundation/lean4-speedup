# C-expansions — concept propositions for the parallel-elaboration contradiction

Protocol v2 step 3 (2026-07-19, iter 75). The named contradiction (TRIZ
input): **commands must observe sequential environment state, yet the wall
is sequential command elaboration on the main thread** (iter 48/49/72:
main occupancy ≈ wall; Mathlib modules run 1.3–1.7 of 16 cores).

Each proposition is a *concept* — undecidable under current knowledge (C-K
sense), generated by the five delta operators ([invention-theory.md](invention-theory.md) §4).
Tag: operator · cheapest discriminating experiment. Selection stays brutal
(playbook rules); generation here is deliberately unfiltered.

Measured ground (iter 75 census, bench/stmtdeps_*): statement-dependency
DAGs are **shallow def-rooted hubs** — Batteries List.Lemmas: 99 % of
statements have zero same-module type deps, statement chain 2/339;
Mathlib Equiv.Basic: chain 3/164, 143 stmt arcs concentrated on 33
targets, ALL top hubs are defs; 50 % of statements dep-free once the top-5
hubs are committed. Renders: docs/assets/arcs-*.svg.

## The catalog

1. **C1 — Wavefront statement elaboration** (status change: promote the
   measured shallow-hub DAG to a design axiom). Elaborate statements
   out-of-order on workers as soon as their (syntactically approximated)
   dep set is committed; sequential fallback on violation
   (Tomasulo/scoreboard transfer). *Experiment: schedule simulator — replay
   a module's per-command main-thread times under wavefront order; predicts
   the ceiling with zero compiler surgery.*
2. **C2 — Abolish "the current environment"** (subtraction — the ether
   move). Commands elaborate against explicit dep-snapshots; the env
   becomes a merge of committed deltas, not a thread-through medium.
   *Experiment: count commands whose elaboration reads env state outside
   their census dep set (instrument `Environment.find?`).*
3. **C3 — Verify-only fast path** (subtraction: "is elaboration necessary
   for checking?"). Statement-level incremental reuse: skip elaboration
   for commands whose syntax hash + dep fingerprints are unchanged; replay
   recorded outputs (rustc red/green at decl granularity, inside modules).
   *Experiment: measure how many decls of a typical Mathlib bump rebuild
   unchanged (statement-hash census across two mathlib4 revisions).*
4. **C4 — The DAG as the artifact** (re-representation — the EDVAC move).
   Emit the statement-dep DAG into the .olean; both downstream consumers
   and the elaborator schedule by it. The representation, not a new
   algorithm, is the deliverable. *Experiment: C1's simulator consumes
   exactly this file — shared cost.*
5. **C5 — Decompose "command order"** (decomposition of the obvious).
   Textual order bundles: (a) name-resolution order, (b) elaboration-
   context order (open/variable/set_option), (c) instance-set order,
   (d) macro/notation-state order. The census shows (a) is shallow; (b) is
   reconstructible per-command from section headers; (c)/(d) writes are
   rare mid-module. Speculation need only respect the rare writers.
   *Experiment: census of (b/c/d)-writing commands per module across the
   corpora.*
6. **C6 — Speculative TC against import instances** (status change:
   "mid-module instances rarely change later statements' TC results" →
   axiom + repair). *Experiment: for hub-defining Mathlib modules, re-run
   each statement's synth queries against import-only instance tables and
   diff the results.*
7. **C7 — One scheduler, all granularities** (unification: two
   explanations for one phenomenon offend). Proof-body async, wavefront
   statements (C1), and lake's module parallelism are the same dependency
   scheduler at three granularities; T3's fission result (module
   boundaries don't matter, dep structure does) is the evidence. Unify:
   one work-stealing scheduler over a decl-level DAG spanning module
   boundaries. *Experiment: extend the simulator across two adjacent
   modules — does merging beat lake's schedule?*
8. **C8 — Elaborate section telescopes once** (subtraction). `variable`
   binders re-elaborate per command; share the elaborated telescope across
   the section's commands. *Experiment: count variable-binder
   re-elaboration cost in a `variable`-heavy Mathlib module profile.*
9. **C9 — Elaboration as a pure memo function** (re-representation).
   Statement elaboration as a pure function of (syntax, dep fingerprints)
   — memoizable across builds and modules; the elaborator becomes a build
   system (Salsa/Shake transfer). Long-horizon version of C3.
10. **C10 — Inline-first instance synthesis** (status change from the
    measured 0.3 ms/inst-mvar orchestration floor, iter 56). Synthesize
    eagerly at creation; fall back to the postpone/resume machinery only
    on failure. *Experiment: count what fraction of pending inst-mvars
    succeed on first attempt in a corpus trace (if ≫90 %, the orchestration
    is almost always pure overhead).*
11. **C11 — Split the env into monotone and versioned parts**
    (decomposition). Constants are monotone (lock-free concurrent commit);
    attributes/options/scoped state are versioned per command. The
    non-monotone part is what actually forces order — isolate it.
    *Experiment: audit which env extensions are written mid-module and how
    often (extends T4/T5's ext-state work).*
12. **C12 — Cross-module statement pipelining** (blind-variance wildcard).
    Elaborate the next module's statements against the current module's
    *predicted* exports (rustc pipelined-rlib transfer). High risk: export
    prediction accuracy unknown. *Experiment: how often does a module's
    export signature set differ from its statement-syntax-derived
    approximation?*

## Priority (Uzzi rule: conventional core, ONE atypical ingredient)

C1 is the entry point: conventional engineering (a scheduler) around one
atypical ingredient (speculating that textual order ≈ no order for
statements), with C4 as its zero-cost representational side effect and C5
as its correctness analysis. C1's simulator is the cheapest next step and
de-risks everything downstream. C3/C9 are the long-horizon re-foundation;
C6/C10/C11 are independently testable sub-bets.

Prior-art check owed before any implementation claim: Lean core's
parallelism roadmap (S. Ullrich's async work covers proof bodies; whether
command-level speculation has been designed/attempted upstream — search
RFCs, zulip).
